## 質問記録

### Q001

- 確定判断: GD001
- 確認したいこと: 挙動差分要約を固定の観点リストにするか、自由記述にするか。
- 確認が必要な理由: この形式が R001 の受け入れ条件になり、レビュアーの確認方法を決めるため。
- 推奨回答: 固定の3観点リスト（変わる判断、変わる成果物構造、影響する後続 phase）を最低ラインとして必須にし、補足は自由記述とする。
- 推奨理由: 3観点は Issue #298 で確定済みの文言であり、固定リストにすることで記載の有無を一目で確認でき、将来の validator または evaluator の検査候補になるため。
- ユーザー回答: 推奨のとおり。

### Q002

- 確定判断: GD002
- 確認したいこと: skill-forge 確認の結果を、PR 説明のどこに、何を書くか。
- 確認が必要な理由: 記録形式が決まらないと、確認の実施をレビュアーが検証できず、R002 の受け入れ条件を定義できないため。
- 推奨回答: PR 説明に固定見出し「skill-forge 確認」を置き、確認した観点（skill 境界、trigger description、本文指示、eval coverage。該当しない観点は「該当なし」）と確認結果（問題なし、または問題と対応）を書く。
- 推奨理由: README が推奨している skill-forge の確認観点をそのまま使うため、新しい語彙を作らず、記録の有無を確認しやすいため。
- ユーザー回答: 推奨のとおり。

### Q003

- 確定判断: GD003
- 確認したいこと: 粒度制約の例外として、同一 PR に含めてよい変更の範囲をどう定めるか。
- 確認が必要な理由: 例外の範囲が曖昧だと制約が形骸化するか、必要な変更まで不自然に分割されるため。
- 推奨回答: 例外を列挙で固定せず、promote 同期は常に同一 PR、他変更種別は分割するとどちらかの PR 単独で検証が fail する不可分な場合だけ許容し、例外時は理由と後続確認先を PR 説明に記録する一般則にする。
- 推奨理由: 列挙方式は変更種別の追加のたびに保守が必要で stale になりやすく、不可分の判定基準は機械的に確認可能で agent-instruction-rules の方針とも合うため。
- ユーザー回答: 推奨のとおり。

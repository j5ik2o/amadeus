# D003: 粒度と User Stories の判断

## 背景

この Intent の価値の受け手は、人間アクターである Reviewer と Maintainer である。

一方、成果物の分解では、Intent と Unit が 1:1 になる。

## 判断

- User Stories を作成する。S001 は Reviewer のレビュー判断価値、S002 は Maintainer の例外承認価値を表す。
- Intent:Unit の 1:1 を例外として認める。Unit は U001 の1件とし、Bolt は B001 と B002 の2件に分解する。
- R004 は固有の相互作用を持たない文書整合制約だが、Reviewer が整合の取れた契約文書を参照できるという前提を支えるため、UC002 へ対応付ける。

## 理由

- レビュー支援契約は人間アクターの価値表現が中心であり、Story を作ると要求から Bolt まで「誰の判断を支えるか」を追跡できる。
- この Intent は単一の steering policy 契約を扱い、複数の価値単位に分けると境界が不自然になる。実施単位は Bolt 2件で分解できている。
- 文書間が矛盾すると UC002 のレビュー判断の根拠が崩れるため、R004 を UC002 の前提として追跡するのが自然である。

## 影響

- 分割を見直す条件: 変更種別が増えて契約の定義元が複数になる場合、または記録の機械検査を導入して相互作用が増える場合は、Unit と Use Case の分割を再検討する。
- `state.json.inception.requiredStoryArtifacts` に S001 と S002 を含める。

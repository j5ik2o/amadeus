# Amadeus 成果物カタログ

このディレクトリは、Amadeus DLC の目的、要求、実装単位、検証状態を追跡するための成果物を置く。

ルートの `CONTEXT.md` と混同しないように、`.amadeus/context.md` は作らない。
システム化の目的、アクター、外部システムは個別ファイルに分ける。

## 基本方針

- 空のプレースホルダーを増やさない。
- 未確認のまま必要な成果物は、未確認であることを本文に書く。
- 任意成果物は、必要性を判断した時点で作る。
- 作らない場合も、作らない判断が追跡に影響するなら `traceability.md` に `なし` と書く。
- 要求から、目的、アクター、外部システムへ戻れるようにする。
- アクターと外部システムは混同しない。
- アクターは、要望、関心、制約、期待価値の源泉、またはシステムとの相互作用に登場する人の役割として扱う。
- ストーリーは、人のアクターのゴール、価値、理由を扱う。
- 外部システムは、システム外部の連携先として扱う。
- 外部システムは `external-systems.md` に定義し、`actors.md` には置かない。
- 成果物の見出しや一覧の概要は、可能な限り主語と動詞を含める。
- 名詞句だけの見出しは、誰の関心か、誰が何をするか、システムが何を提供するかを曖昧にするため避ける。
- ユースケースは、人のアクターと外部システムを含む相互作用を扱う。
- ユースケースは、少なくともアクターまたは外部システムのどちらかを参照する。
- ユースケースは、アクターまたは外部システムとシステムの対話を、システム境界の外側から内側へ叙述的に書く。
- ユースケースを「できる」だけの要求文として書かない。
- ユースケースは、用語集、ドメインモデル、外部システムで定義した語を使って書く。
- ユースケースの名詞は、境界またはエンティティ候補として読める語にする。
- ユースケースの動詞は、制御候補として読める処理にする。
- 「ページ」「データ」「処理」のような一般語だけでユースケースの中心語を表さない。
- ユースケースには、境界、制御、エンティティの候補を記録する。
- 境界、制御、エンティティの候補は実装設計そのものではなく、ロバストネス分析の入力として扱う。
- ユースケースには、候補ごとの責務も記録する。
- 責務候補は実装設計そのものではなく、設計へ進むための分析結果として扱う。
- タスクは要求を必ず参照する。
- タスクは原則としてユースケースを参照する。
- 相互作用がない内部作業では、ユースケースを参照しない理由を `traceability.md` に書く。
- タスクはストーリーを直接参照しない。

## ストーリー、ユースケース、タスクの追跡

タスクの親は常にボルトである。
ユースケースはタスクの親ではなく、タスクが何を満たすかを示す参照先である。

次の3パターンを許可する。

1. `User Story <- Use Case <- Task`
   ストーリーのゴールをユースケースの相互作用手順に落とし、その手順をタスクで実現する。

2. `Use Case <- Task`
   ユーザーストーリーがないシステム間連携などで、ユースケースをタスクで実現するケース。

3. `Requirement <- Task`
   リファクタ、内部品質改善、設定変更など、アクターまたは外部システムとの相互作用がない内部作業。
   この場合は、ユースケースを参照しない理由を `traceability.md` に書く。

タスクは要求を必ず参照する。
タスクは原則としてユースケースを参照する。
内部作業でユースケースを参照しない場合は、理由を `traceability.md` に残す。
タスクはストーリーを直接参照しない。

ユースケースは要求を必ず参照する。
ユースケースはストーリーを任意で参照できる。
ユースケースはアクターまたは外部システムを必ず参照する。

## ユースケース内の項目

| 項目 | 必須度 | 作成タイミング | 役割 |
|---|---|---|---|
| システム境界 | 必須 | ユースケース作成時 | 外側と内側の対話境界を明示する |
| 事前条件 | 必須 | ユースケース作成時 | フロー開始前に成り立つ条件を明示する |
| 基本フロー | 必須 | ユースケース作成時 | 通常の相互作用手順を叙述的に示す |
| 代替フロー | 任意 | 分岐や例外がある時 | 基本フローから外れる相互作用を示す |
| 事後条件 | 必須 | ユースケース作成時 | フロー終了後に成り立つ条件を明示する |
| BCE候補 | 必須 | ユースケース作成時 | 境界、制御、エンティティの候補を示す |
| 責務候補 | 必須 | ユースケース作成時 | 候補ごとの責務を示す |
| 役割ステレオタイプ | 任意 | BCEだけで責務判断が難しい時 | 情報保持役、サービス提供役、構造役、インターフェイス役、調整役、制御役で見直す |
| シーケンス図 | 任意 | 動的な責務割り当てを確認したい時 | アクター、境界、制御、エンティティのメッセージを確認する |

責務候補は、どの候補が何を判断し、何を保持し、何を依頼するかを明らかにする。
責務候補は確定設計ではなく、設計へ進むための分析結果として扱う。

## インテント間の依存

`intents.md` には、インテント一覧とインテント間の依存関係を書く。
インテントが別のインテントの成果を前提にする場合は、一覧の `依存` に先行インテントの ID を書く。
依存がない場合は `なし` と書く。
インテント間依存は目的単位の前提関係を示す。
要求間依存は受け入れ条件の前提関係を示す。
ストーリー間依存はユーザー価値表現の前提関係を示す。
ユースケース間依存は相互作用手順の前提関係を示す。
ユニット間依存は価値単位の前提関係を示す。
ボルト間依存は実装と検証の反復順序を示す。
タスク間依存は具体的な作業順序を示す。
判断間依存は判断の前提関係を示す。
どの依存も、何を作るか、何を確認するかの説明の代わりにはならない。

## 要求間の依存

`requirements.md` には、要求間の依存関係を書く。
要求が別の要求の成立や成果を前提にする場合は、一覧の `依存` に先行要求の ID を書く。
依存がない場合は `なし` と書く。
要求間依存は、受け入れ条件の前提関係と検証順序を示す。
要求間依存は、ユニット、ボルト、タスクの依存へ反映される。

## ストーリー間の依存

`user-stories.md` には、ストーリー間の依存関係を書く。
ストーリーが別のストーリーのゴール達成や成果を前提にする場合は、一覧の `依存` に先行ストーリーの ID を書く。
依存がない場合は `なし` と書く。
ストーリー間依存は、ユーザー価値表現の前提関係を示す。
ストーリーは任意成果物であるため、ストーリーがない作業ではストーリー間依存も作らない。

## ユースケース間の依存

`use-cases.md` には、ユースケース間の依存関係を書く。
ユースケースが別のユースケースで得られる状態や成果を前提にする場合は、一覧の `依存` に先行ユースケースの ID を書く。
依存がない場合は `なし` と書く。
ユースケース間依存は、相互作用手順の前提関係を示す。
ユースケース間依存は、ユニット、ボルト、タスクの依存へ反映される。

## ユニット間の依存

`units.md` には、ユニット間の依存関係を書く。
ユニットが別のユニットの価値や成果を前提にする場合は、一覧の `依存` に先行ユニットの ID を書く。
依存がない場合は `なし` と書く。
ユニット間依存は価値単位の前提関係を示す。
ボルト間依存は実装と検証の反復順序を示す。
タスク間依存は具体的な作業順序を示す。
どの依存も、何を作るか、何を確認するかの説明の代わりにはならない。

## タスク内の項目

タスクは、識別子付きチェックリストの形式を保つ。
タスクの見出しには、何を完了させるかが分かる短い文を書く。
詳細は、見出しの下に箇条書きで書く。

```md
- [ ] T001: パスワード再設定要求の入力を定義する
  - 作業:
    - 利用者が入力する識別子を定義する。
    - 入力値を受け付けるシステム境界を確認する。
  - 要求: R001
  - ユースケース: UC001
  - 依存: なし
  - 証拠: 未登録
```

タスクは、具体的に何を作るか、何を確認するかを `作業` に書く。
ただし、関数名、ファイル名、実装順序の細部まで固定しない。

タスク間に順序制約がある場合は、`依存` に先行タスクの ID を書く。
依存がない場合は `なし` と書く。
`依存` は実行順序を表すだけであり、タスクの内容説明の代わりにはならない。
他のボルトにあるタスクへ依存する場合は、`B001/T001` のようにボルト ID とタスク ID を組み合わせて書く。
同じボルト内のタスクへ依存する場合は、タスク ID だけを書いてよい。

## ボルト間の依存

`bolts.md` には、ボルト間の依存関係を書く。
ボルトが別のボルトの成果を前提にする場合は、一覧の `依存` に先行ボルトの ID を書く。
依存がない場合は `なし` と書く。
ボルト間依存は粗い実行順序を示し、タスク間依存は具体的な作業順序を示す。
どちらも、何を作るか、何を確認するかの説明の代わりにはならない。

## 判断間の依存

`decisions.md` には、判断一覧と判断間の依存関係を書く。
判断が別の判断の採用や前提を必要とする場合は、一覧の `依存` に先行判断の ID を書く。
依存がない場合は `なし` と書く。
判断間依存は、設計上の前提関係と判断順序を示す。
個別判断の背景、選択肢、理由は `decisions/<decision-id>.md` に書く。

## インテント固有ドメインモデル

`.amadeus/domain-model.md` は、Amadeus DLC 全体のドメインモデルを扱う。
全体モデルには、各インテント固有モデルへの索引を置く。
インテント固有の概念関係や契約が必要な場合は、インテント配下の `domain/` に置く。

`domain/model.md` には、そのインテントで使う概念、関係、ライフサイクル、集約候補を書く。
`domain/contracts.md` には、そのインテントで守る事前条件、不変条件、事後条件と根拠を書く。
未確定事項や昇格候補は `domain-notes.md` に残す。

## 識別子

| 種別 | 形式 | 例 |
|---|---|---|
| 目的 | `OBJnnn` | `OBJ001` |
| アクター | `ACTnnn` | `ACT001` |
| 外部システム | `EXTnnn` | `EXT001` |
| 要求 | `Rnnn` | `R001` |
| ストーリー | `Snnn` | `S001` |
| ユースケース | `UCnnn` | `UC001` |
| ユニット | `Unnn` | `U001` |
| ボルト | `Bnnn` | `B001` |
| タスク | `Tnnn` | `T001` |
| 事前条件 | `PREnnn` | `PRE001` |
| 不変条件 | `INVnnn` | `INV001` |
| 事後条件 | `POSTnnn` | `POST001` |

契約識別子は、`domain/contracts.md` 内でだけ採番する。
`PREnnn` は事前条件、`INVnnn` は不変条件、`POSTnnn` は事後条件を表す。

- [ ] 要求が参照する目的を確認した。
- [ ] 要求が関係するアクターを確認した。
- [ ] 要求が関係する外部システムを確認した。
- [ ] ストーリーが必要か判断した。
- [ ] ストーリーを作る場合は、`actors.md` にアクターが定義されていることを確認した。
- [ ] ユースケースが必要か判断した。
- [ ] ユースケースを作る場合は、アクターまたは外部システムへの参照があることを確認した。
- [ ] 受け入れ状態を記録した。
- [ ] 追跡表を更新した。

## ボルト作成チェック

- [ ] ボルトが参照する要求を確認した。
- [ ] ボルトが参照するユニットを確認した。
- [ ] タスクが参照するユースケース、またはユースケースを参照しない理由を確認した。
- [ ] タスクを識別子付きで列挙した。
- [ ] 各タスクが要求を参照していることを確認した。
- [ ] 各タスクがユースケースを参照しているか、内部作業として理由を `traceability.md` に残していることを確認した。

# Amadeus 成果物カタログ

このディレクトリは、Amadeus DLC の目的、要求、実装単位、検証状態を追跡するための成果物を置く。

ルートの `CONTEXT.md` と混同しないように、`.amadeus/context.md` は作らない。
システム化の目的、アクター、外部システムは個別ファイルに分ける。

## 基本方針

- 空のプレースホルダーを増やさない。
- 未確認のまま必要な成果物は、未確認であることを本文に書く。
- 任意成果物は、必要性を判断した時点で作る。
- 作らない場合も、作らない判断が追跡に影響するなら `traceability.md` に `なし` と書く。
- 要求から、目的、アクター、外部システムへ戻れるようにする。
- アクターと外部システムは混同しない。
- アクターは、要望、関心、制約、期待価値の源泉、またはシステムとの相互作用に登場する人の役割として扱う。
- ストーリーは、人のアクターのゴール、価値、理由を扱う。
- 外部システムは、システム外部の連携先として扱う。
- 外部システムは `external-systems.md` に定義し、`actors.md` には置かない。
- 成果物の見出しや一覧の概要は、可能な限り主語と動詞を含める。
- 名詞句だけの見出しは、誰の関心か、誰が何をするか、システムが何を提供するかを曖昧にするため避ける。
- ユースケースは、人のアクターと外部システムを含む相互作用を扱う。
- ユースケースは、少なくともアクターまたは外部システムのどちらかを参照する。
- ユースケースは、アクターまたは外部システムとシステムの対話を、システム境界の外側から内側へ叙述的に書く。
- ユースケースを「できる」だけの要求文として書かない。
- ユースケースは、用語集、ドメインモデル、外部システムで定義した語を使って書く。
- ユースケースの名詞は、境界またはエンティティ候補として読める語にする。
- ユースケースの動詞は、制御候補として読める処理にする。
- 「ページ」「データ」「処理」のような一般語だけでユースケースの中心語を表さない。
- ユースケースには、必要に応じて境界、制御、エンティティの候補を記録する。
- 境界、制御、エンティティの候補は実装設計そのものではなく、ロバストネス分析の入力として扱う。
- タスクは要求を必ず参照する。
- タスクはユースケースを必ず参照する。
- タスクはストーリーを直接参照しない。

## ストーリー、ユースケース、タスクの追跡

タスクの親は常にボルトである。
ユースケースはタスクの親ではなく、タスクが何を満たすかを示す参照先である。

次の2パターンを許可する。

1. `User Story <- Use Case <- Task`
   ストーリーのゴールをユースケースの相互作用手順に落とし、その手順をタスクで実現する。

2. `Use Case <- Task`
   ユーザーストーリーがないシステム間連携などで、ユースケースをタスクで実現するケース。

タスクは要求を必ず参照する。
タスクはユースケースを必ず参照する。
タスクはストーリーを直接参照しない。

ユースケースは要求を必ず参照する。
ユースケースはストーリーを任意で参照できる。
ユースケースはアクターまたは外部システムを必ず参照する。

## ユースケース内の項目

| 項目 | 必須度 | 作成タイミング | 役割 |
|---|---|---|---|
| システム境界 | 必須 | ユースケース作成時 | 外側と内側の対話境界を明示する |
| 事前条件 | 必須 | ユースケース作成時 | フロー開始前に成り立つ条件を明示する |
| 基本フロー | 必須 | ユースケース作成時 | 通常の相互作用手順を叙述的に示す |
| 代替フロー | 任意 | 分岐や例外がある時 | 基本フローから外れる相互作用を示す |
| 事後条件 | 必須 | ユースケース作成時 | フロー終了後に成り立つ条件を明示する |
| BCE候補 | 必須 | ユースケース作成時 | 境界、制御、エンティティの候補を示す |
| 責務候補 | 必須 | ユースケース作成時 | 候補ごとの責務を示す |
| 役割ステレオタイプ | 任意 | BCEだけで責務判断が難しい時 | 情報保持役、サービス提供役、構造役、インターフェイス役、調整役、制御役で見直す |
| シーケンス図 | 任意 | 動的な責務割り当てを確認したい時 | アクター、境界、制御、エンティティのメッセージを確認する |

責務候補は、どの候補が何を判断し、何を保持し、何を依頼するかを明らかにする。
責務候補は確定設計ではなく、設計へ進むための分析結果として扱う。

## グローバル成果物

| 成果物 | 必須度 | 作成タイミング | 役割 | 未作成時の扱い |
|---|---|---|---|---|
| `README.md` | 必須 | `.amadeus/` 初期化時 | 成果物の役割、必須度、作成タイミングを定義する | 未作成不可 |
| `objective.md` | 必須 | `.amadeus/` 初期化時 | システム化の目的、期待価値、成功指標を扱う | 未確認なら状態を書く |
| `actors.md` | 必須 | `.amadeus/` 初期化時 | 要求の根拠やシステムとの相互作用に登場する人の役割を扱う | 未確認なら状態を書く |
| `external-systems.md` | 任意 | 外部システムとの連携が出た時 | システム外部の連携先と接点を扱う | 外部連携がなければ作らない |
| `glossary.md` | 必須 | `.amadeus/` 初期化時 | 用語定義、避ける語、禁止ワードを扱う | 未作成不可 |
| `domain-model.md` | 必須 | `.amadeus/` 初期化時 | 概念間の関係、不変条件、ライフサイクル、集約候補を扱う | 未確認なら状態を書く |
| `decisions/` | 任意 | 判断を残す必要がある時 | 全体構造に関わる判断を記録する | 判断がなければ作らない |

## インテント成果物

| 成果物 | 必須度 | 作成タイミング | 役割 | 未作成時の扱い |
|---|---|---|---|---|
| `intents/<intent-id>/intent.md` | 必須 | インテント作成時 | インテントの目的、成功条件、範囲を扱う | 未作成不可 |
| `requirements.md` | 必須 | 最初の要求を作る時 | 要求一覧と受け入れ状態を扱う | 要求がなければ作らない |
| `acceptance.md` | 必須 | 最初の要求を作る時 | 要求の受け入れ状態と証拠を扱う | 要求がなければ作らない |
| `traceability.md` | 必須 | 最初の要求を作る時 | 目的、アクター、外部システム、要求、ストーリー、ユースケース、ユニット、ボルト、タスクの対応を扱う | 要求がなければ作らない |
| `requirements/<requirement-id>.md` | 任意 | 要求単位の説明、受け入れ条件、背景参照が必要な時 | 個別要求を扱う | 一覧で十分なら作らない |
| `stories.md` | 任意 | ストーリーが必要な時 | アクターを明示したストーリー一覧を扱う | 不要なら `traceability.md` に `なし` と書く |
| `stories/<story-id>.md` | 任意 | ストーリー単位の説明が必要な時 | アクター、価値、受け入れ条件を持つ個別ストーリーを扱う | 一覧で十分なら作らない |
| `use-cases.md` | 必須 | 最初のタスクを作る時 | アクターまたは外部システムを明示したユースケース一覧を扱う | タスクがなければ作らない |
| `use-cases/<use-case-id>.md` | 必須 | タスクが参照するユースケースを作る時 | アクター、外部システム、事前条件、基本フロー、代替フロー、事後条件を扱う | タスクから参照しないなら作らない |
| `units.md` | 必須 | 最初のユニットを作る時 | ユニット一覧を扱う | ユニットがなければ作らない |
| `units/<unit-id>.md` | 任意 | ユニット単位の説明が必要な時 | 個別ユニットを扱う | 一覧で十分なら作らない |
| `bolts.md` | 必須 | 最初のボルトを作る時 | ボルト一覧を扱う | ボルトがなければ作らない |
| `bolts/<bolt-id>/bolt.md` | 必須 | ボルト作成時 | ボルトの範囲、対象ユニット、完了条件を扱う | ボルトがなければ作らない |
| `bolts/<bolt-id>/tasks.md` | 必須 | ボルト作成時 | タスクを識別子付きチェックリストで扱う | ボルトがなければ作らない |
| `bolts/<bolt-id>/notes.md` | 任意 | ボルト固有のメモが必要な時 | 実装中の判断、注意点、未確定事項を扱う | メモがなければ作らない |
| `bolts/<bolt-id>/pr.md` | 任意 | プルリクエストができた時 | プルリクエストのリンクと要約を扱う | 未作成の間は作らない |
| `bolts/<bolt-id>/test-results.md` | 任意 | テスト結果が出た時 | 受け入れ証拠として使うテスト結果を扱う | 未実行の間は作らない |
| `domain-notes.md` | 任意 | インテント固有のモデル上の発見がある時 | 未確定事項、昇格候補を扱う | 発見がなければ作らない |
| `terminology-notes.md` | 任意 | インテント固有の未確定語がある時 | 未確定語、提案語、用語確認事項を扱う | 未確定語がなければ作らない |
| `actor-notes.md` | 任意 | インテント固有のアクター上の発見や未確定事項がある時 | アクター候補、相互作用上の疑問、昇格候補を扱う | 発見がなければ作らない |
| `decisions/` | 任意 | インテント固有の判断を残す必要がある時 | インテント内の判断を記録する | 判断がなければ作らない |

## 初期化チェック

- [ ] システムの目的を確認した。
- [ ] アクターを確認した。
- [ ] 外部システムが必要か判断した。
- [ ] 用語定義、避ける語、禁止ワードを確認した。
- [ ] ドメインモデルの初期状態を確認した。

## 要求作成チェック

- [ ] 要求が参照する目的を確認した。
- [ ] 要求が関係するアクターを確認した。
- [ ] 要求が関係する外部システムを確認した。
- [ ] ストーリーが必要か判断した。
- [ ] ストーリーを作る場合は、`actors.md` にアクターが定義されていることを確認した。
- [ ] ユースケースが必要か判断した。
- [ ] ユースケースを作る場合は、アクターまたは外部システムへの参照があることを確認した。
- [ ] 受け入れ状態を記録した。
- [ ] 追跡表を更新した。

## ボルト作成チェック

- [ ] ボルトが参照する要求を確認した。
- [ ] ボルトが参照するユニットを確認した。
- [ ] タスクが参照するユースケースを確認した。
- [ ] タスクを識別子付きで列挙した。
- [ ] 各タスクが要求を参照していることを確認した。
- [ ] 各タスクがユースケースを参照していることを確認した。
- [ ] 各タスクがストーリーを直接参照していないことを確認した。
- [ ] プルリクエスト記録が必要か判断した。
- [ ] テスト結果記録が必要か判断した。
- [ ] ボルト固有メモが必要か判断した。
